## Зачем нормализация — одна метафора

Представь Excel. Один большой лист, всё подряд:

| user_name | category | category_emoji | amount |
|-----------|----------|----------------|--------|
| Artem     | Продукты | 🛒             | 150    |
| Artem     | Продукты | 🛒             | 200    |
| Ivan      | Продукты | 🛒             | 90     |

"Продукты" и "🛒" повторяются в каждой строке.
Захочешь поменять эмодзи — придётся менять в 1000 строках.
Это **ненормализованная база**.

Нормализация — устранение повторений и правильное разделение данных по таблицам.

---

## Три правила (1NF → 2NF → 3NF)

### 1NF — Первая нормальная форма
**Правило:** каждая ячейка содержит одно атомарное значение. Никаких списков внутри.

```
❌ Нарушение:
| user  | categories             |
| Artem | Продукты, Транспорт    |  ← список в одной ячейке

✅ В проекте — соблюдено:
каждая строка transactions = одна операция, одно значение amount
```

---

### 2NF — Вторая нормальная форма
**Правило:** каждый столбец зависит от **всего** первичного ключа, а не от его части.

Актуально когда ключ составной (из двух колонок). Если ключ простой — 2NF выполняется автоматически.

```
❌ Нарушение (составной ключ: order_id + product_id):
| order_id | product_id | product_name | quantity |
product_name зависит только от product_id — нарушение.

✅ В проекте — исправлено через отдельную таблицу categories:
"Продукты" хранится ОДИН РАЗ в categories.name
transactions хранит только category_id (ключ), не название
```

---

### 3NF — Третья нормальная форма
**Правило:** не-ключевые столбцы не зависят друг от друга (транзитивная зависимость).

```
❌ Нарушение:
| tx_id | category_id | category_name | category_emoji |
category_name зависит от category_id, а не от tx_id — транзитивность.

✅ В проекте — соблюдено:
в transactions нет никаких данных о категории кроме category_id.
```

## Схема проекта — визуально

```
┌─────────────┐          ┌──────────────────┐          ┌────────────────────┐
│    users    │          │   transactions   │          │    categories      │
│─────────────│          │──────────────────│          │────────────────────│
│ id (PK)     │◄─────────│ user_id (FK)     │          │ id (PK)            │
│ username    │          │ category_id (FK) │─────────►│ name  (UNIQUE)     │
│ first_name  │          │ amount           │          │ type  (income/exp) │
│ created_at  │          │ created_at       │          │ emoji              │
└─────────────┘          └──────────────────┘          └────────────────────┘
```

`transactions` — центральная таблица.
Она ничего не знает о пользователе или категории напрямую — только хранит ключи (FK).
Всё остальное получается через JOIN при запросе.

## ACID — четыре гарантии SQLite

| Буква | Слово        | Простыми словами                                                     |
|-------|--------------|----------------------------------------------------------------------|
| **A** | Atomicity    | Операция либо выполнилась целиком, либо не выполнилась вообще       |
| **C** | Consistency  | База всегда в правильном состоянии: CHECK(amount > 0) не даст -150  |
| **I** | Isolation    | Два одновременных запроса не мешают друг другу                      |
| **D** | Durability   | После COMMIT данные сохранены даже если упадёт свет                 |

### Как это реализовано в проекте (`database.py`):

```python
await db.execute("PRAGMA journal_mode=WAL")
# WAL = Write-Ahead Log
# Читатели не блокируют писателей — важно при нескольких пользователях

await db.execute("PRAGMA foreign_keys=ON")
# Без этого SQLite ИГНОРИРУЕТ внешние ключи по умолчанию (!)

await db.execute("PRAGMA synchronous=NORMAL")
# Баланс между скоростью и надёжностью записи на диск
```

## Как проверить уровень нормализации — 3 вопроса

```
1. Есть ли в ячейках списки или несколько значений?
   Да → нарушена 1NF

2. Есть ли столбцы, которые зависят только от ЧАСТИ составного ключа?
   Да → нарушена 2NF

3. Есть ли столбцы, которые зависят от другого не-ключевого столбца?
   Да → нарушена 3NF
```

Если на все три вопроса ответ **"нет"** — база в 3NF. ✅
